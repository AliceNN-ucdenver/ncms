# Fitness Functions: Keeping the Architecture Honest

**Status:** Active
**Audience:** Engineering leadership, architecture review, CIO
**Date:** 2026-04-16

---

## Executive summary

NCMS runs four machine-checked **fitness functions** on every test run.
They do not test behavior; they test that the *architecture itself*
hasn't eroded.  Three minutes of added CI time converts architectural
decisions from "something a senior engineer remembers" into "something
a failing build enforces."

This matters more in 2026 than it did in 2020.  AI coding assistants
now produce changes at a pace that outstrips human review.  Without
machine-enforced structural invariants, architecture drifts silently
— the system keeps working, tests keep passing, and three months later
a "minor change" takes two weeks because the boundaries have been
quietly eaten away.  Fitness functions close that gap.

---

## The problem they solve

A typical enterprise codebase accumulates three kinds of decay:

1. **Complexity drift.**  Methods grow from 50 lines to 500 one commit
   at a time.  No single change looks bad, but the sum becomes
   unmaintainable.
2. **Boundary erosion.**  A layer that was supposed to be isolated
   gets a "just this one import" added under pressure.  Six months
   later every module imports from every other module and the
   modularity claim is fiction.
3. **Dead code accumulation.**  Features get deprecated, branches get
   abandoned, and the orphaned code continues to load, ship, and confuse
   search results in AI-assisted editing.

Manual review catches **some** of each.  It catches far less when
changes are being generated by an AI at 10× human velocity, because
reviewers start skimming instead of reading.  Fitness functions are the
load-bearing part of the review that reviewers stop doing when the
pipeline is fast.

## What's in place today

The four functions live under `tests/architecture/` and run with the
regular test suite (~0.4 s total):

### 1. Complexity gate — "no method may silently become unmaintainable"

- **What it enforces:** every method in `src/ncms/application/` stays
  at cyclomatic complexity grade C or better.  A D-grade method (21+
  branches) fails the build.
- **Why it matters:** before Phase 0, one file (`memory_service.py`)
  was 3,522 lines with seven D-grade methods.  After Phase 0 it's a
  1,220-line composition root that delegates to five focused pipelines.
  The gate guarantees that subsequent changes — including AI-generated
  refactors — cannot re-merge it.  Any attempt to "simplify" by
  inlining back into the orchestrator fails fast.
- **Escape hatch:** an explicit `ALLOWLIST` in the test file with a
  comment explaining why.  Empty today by design.

### 2. Import boundaries — "layers don't leak"

- **What it enforces:** three invariants about module dependencies:
  1. The domain layer (`ncms/domain/`) imports nothing from
     `application/` or `infrastructure/`.  This preserves Clean
     Architecture's inner-ring purity.
  2. The five extracted pipelines (`scoring`, `retrieval`,
     `enrichment`, `ingestion`, `traversal`) may not import each
     other.  They compose through `MemoryService`.
  3. Pipelines may not import their own composition root, eliminating
     circular ownership.
- **Why it matters:** the whole point of the Phase 0 refactor was
  that each upcoming feature (temporal query boosting, preference
  extraction, dense embeddings, session storage) lands in exactly one
  pipeline.  The moment one pipeline imports another, that contract
  breaks — a change to retrieval silently forces a change to scoring.
  The test prevents the breach at source-code parse time; it doesn't
  need to run anything.

### 3. Dead code detector — "the corpse must go"

- **What it enforces:** Vulture at 80% confidence finds no unused
  symbols in `src/ncms/`, excluding the five deliberate false positives
  recorded in `.vulture_whitelist.py` (signal-handler `frame`
  parameters and Prometheus no-op `amount` kwargs).
- **Why it matters:** AI coding tools are confused by dead code.
  They cite it as context, suggest edits to it, and occasionally
  resurrect it.  Keeping the corpus clean directly improves the
  quality of AI-assisted changes.

### 4. Inherited baseline — ruff and radon run on every commit

- `ruff` enforces the style guide and catches the 80% of obvious
  issues (unused imports, sorted imports, unused variables) without
  human effort.
- `radon` provides the metrics the complexity gate operates on;
  developers can also run it locally (`uv run radon cc src/ncms/`) to
  see trends.

## Why this matters for AI-assisted development

Traditional code review assumes a roughly even pace: humans write, other
humans review, the review has capacity to consider structure as well
as correctness.  AI-assisted development breaks that assumption in two
ways:

**Throughput.** An engineer directing an AI assistant can produce
5–10× the diff volume of a pre-AI engineer.  Architecture review that
scaled to one expert-hour per PR no longer scales to ten expert-hours
per PR.  Fitness functions absorb the structural-integrity part of
review, freeing human review to focus on intent, security, and domain
correctness — the things a machine cannot verify.

**Context limits.** AI agents see only what fits in their context
window.  They do not know the unwritten rule that "scoring and
retrieval are independent pipelines" unless that rule is expressed as
a test the agent itself runs.  Our fitness functions *are* that
expression.  When the agent violates the rule, the test fails in its
own feedback loop — it self-corrects without human intervention.

The combination — machine throughput plus machine-enforced invariants
— is what makes AI-assisted architecture-heavy work sustainable.
Without the second half, the first half produces technical debt faster
than humans can retire it.

## What fitness functions do *not* do

To set the expectation honestly:

- **They don't prove the architecture is correct.**  They prove it
  hasn't changed.  If the original design was wrong, the tests will
  happily enforce the wrong design.
- **They don't replace code review.**  They replace the *structural*
  part of code review — the part reviewers skim when they're tired.
  Intent, security, and domain correctness still need human eyes.
- **They don't catch behavioral regressions.**  That's what the other
  845 tests are for.  Fitness functions and behavioral tests are
  complementary, not substitutes.

## Roadmap

Short term (1-2 weeks):

- Wire the fitness functions into CI as a gate on main (they already
  run in `pytest`; this just needs the GitHub Action to fail on them).
- Add a fitness function for test quality: no test asserts raw
  collection counts of user-extensible things (tool lists, enum
  members, CLI commands).  The rule caught two stale tests during
  Phase 0 cleanup; codifying it prevents the pattern recurring.

Medium term (1-2 months):

- Coverage threshold as a fitness function: no module below 70%
  branch coverage for application code.  Codifies the current implicit
  bar.
- "Public API" gate: `memory_service.py` public methods may not shrink
  without a migration note, to prevent accidentally breaking
  integrations.

Longer term:

- Performance budget fitness functions.  Assert that search p95 stays
  under N ms on a reference dataset; fail the build on regression.
  Requires a benchmark harness we already have for retrieval quality
  (SciFact, LongMemEval) — the infrastructure is there.

- Run with vulture: `uv run --with vulture pytest tests/architecture/`
- All four fitness functions together: ~0.4 seconds.
