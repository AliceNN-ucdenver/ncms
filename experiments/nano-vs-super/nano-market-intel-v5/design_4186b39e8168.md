## 2. API Endpoint Specifications  

| Route | Method | Request Type | Response Type | Description | Success Status | Error Mapping |
|-------|--------|--------------|---------------|-------------|----------------|--------------|
| `/auth/register` | `POST` | `RegisterDto` (email, password, role) | `UserResponse` (id, email, role) | Register a new user, store password with **bcrypt (cost ≥ 12)**, emit `UserCreated` event | **201** | **400** if validation fails, **409** if email exists |
| `/auth/login` | `POST` | `LoginDto` (email, password) | `TokensResponse` (accessToken, refreshToken, expiresIn) | Verify credentials, issue short‑lived **JWT (≤ 15 min)**, generate rotating **refresh token** (HTTP‑only cookie) | **200** | **401** if credentials invalid |
| `/auth/refresh` | `POST` | `RefreshDto` (refreshToken) | `TokensResponse` | Exchange refresh token, rotate both tokens, add new JWT to revocation index | **200** | **401** if token expired/invalid, **403** if revoked |
| `/auth/logout` | `POST` | – | – | Add current JWT to revocation store, clear refresh‑token cookie | **204** | **401** if no valid session |
| `/movies` | `GET` | query params (title, genre…) | `MovieResponse[]` | Read‑only public movie catalog (embedded cast) | **200** | **500** if DB error |
| `/reviews` | `POST` | `CreateReviewDto` (movieId, rating, comment, role) | `ReviewResponse` | Write review – **role‑based** (`reviewer`/`admin`) enforced by RBAC middleware | **201** | **403** if role not allowed |
| `/reviews` | `GET` | query (movieId, page) | `PaginatedReviews` | Retrieve reviews (paginated) | **200** | **400** if params malformed |

## 4. Authentication Middleware Implementation  

### 4.1 JWT Service (`jwtStrategy.ts`)  

```ts
// src/auth/jwtStrategy.ts
import jwt from 'jsonwebtoken';
import { IUser } from '../db/user.model';
import { config } from '@/core/config';
import { generateRefreshToken } from '@/auth/authService';
import { addRevocationEntry } from '@/auth/revocation.service';

const signAccessToken = (payload: object): string =>
  jwt.sign(payload, config.JWT_PRIVATE_KEY, {
    algorithm: 'RS256',
    expiresIn: config.JWT_ACCESS_EXPIRES_IN, // e.g., '15m'
    keyid: config.JWT_KEY_ID,
  });

export async function issueTokens(user: IUser): Promise<{
  accessToken: string;
  refreshToken: string;
}> {
  const access = signAccessToken({
    sub: user.id,
    role: user.role,
  });

  const refresh = generateRefreshToken(user.id);
  await addRevocationEntry(user.id, access); // store current access token for revocation check

  return { accessToken: access, refreshToken: refresh };
}
```

### 4.2 RBAC Middleware (`rbacMiddleware.ts`)  

```ts
// src/auth/rbacMiddleware.ts
import { Request, Response, NextFunction } from 'express';
import { JwtPayload } from '@/models/jwt.interfaces';
import { config } from '@/core/config';
import jwt from 'jsonwebtoken';

export function requireRole(...allowed: string[]) {
  return async (req: Request, res: Response, next: NextFunction) => {
    const token = req.headers.authorization?.split(' ')[1];
    if (!token) return res.status(401).json({ message: 'Missing token' });

    try {
      const decoded = jwt.verify(token, config.JWT_PUBLIC_KEY, {
        algorithms: ['RS256'],
        keyid: config.JWT_KEY_ID,
      }) as JwtPayload;

      if (!allowed.includes(decoded.role)) {
        return res.status(403).json({ message: 'Insufficient role' });
      }
      // Attach user info to request for downstream handlers
      (req as any).user = decoded;
      next();
    } catch (err) {
      return res.status(401).json({ message: 'Invalid token' });
    }
  };
}
```

### 4.3 Session Management  

*Stateless*: No server‑side session store. The **revocation service** stores the *current* access token’s `jti` (JWT ID) to enable immediate invalidation on logout or token rotation.  

```ts
// src/auth/revocation.service.ts
import { redis } from '@/core/redisClient'; // redis client wrapper

export async function addRevocationEntry(userId: string, token: string) {
  const jti = (jwt.decode(token) as any)?.jti;
  if (!jti) return;
  // keep entry for up to 30 min (covers token lifespan)
  await redis.set(`revoked:${jti}`, '1', 'EX', 1800);
}

// Token verification wrapper used by middleware
export async function isTokenRevoked(jti: string): Promise<boolean> {
  return Boolean(await redis.get(`revoked:${jti}`));
}
```

### 7.2 Central Error Formatter  

```ts
// src/middleware/errorHandler.ts (continued)
export const errorHandler = (
  err: ApiError | Error,
  _req: Request,
  _res: Response,
  _next: NextFunction,
) => {
  const status = (err as ApiError).statusCode || 500;
  const message = (err as ApiError).message || 'Internal Server Error';
  const details = (err as ApiError).details;

  // Log – structured JSON for audit (GDPR Art. 30)
  console.error('[ERROR]', {
    timestamp: new Date().toISOString(),
    path: _req.originalUrl,
    status,
    message,
    ...details,
  });

  _res.status(status).json({
    error: {
      message,
      ...(details && { details }),
    },
  });
};
```

### 7.3 Consistency  

All controllers wrap async logic in `try/catch` and re‑throw an `ApiError`. This mirrors **R9‑b** (ROI‑focused observability) by ensuring every failure is captured in the monitoring pipeline.

## 10. Design Rationale & Research Traceability  

### 10.1 Patent Landscape Alignment  

- **Patents examined**: US20260067075A1 (handle‑based auth), US20260067072A1 (quantum‑random mobile auth), US20260095322A1, US20260058818A1 (contactless‑card key‑selection).  
- **Design choices that stay clear of claim overlap**:  
  - We use **standard RSA‑256 JWT signing** rather than the patented *handle‑based* key record system (US20260067075A1).  
  - No implementation of **quantum‑random number generation** for authentication challenges; we rely on OS‑provided CSPRNG, avoiding the scope of US20260067072A1.  
  - Contactless‑card key‑selection is not part of our roadmap; we focus on **web/mobile JWT** flows.  

*Freedom‑to‑operate* assessment (R12, R13) concluded that the only relevant patent families are **cryptographic key management** and **contactless‑card authentication**, both of which our design sidesteps by using industry‑standard JWT and Redis‑based revocation.  

### 10.2 JTBD Alignment  

- Primary job‑to‑be‑Done (R8) = “**secure access to sensitive systems while maintaining regulatory compliance and minimizing user friction**.”  
- **Design responses**:  
  - **Compliance‑first engine** (`authService.ts`) enforces NIST SP 800‑63B‑4 (R2).  
  - **Adaptive AI risk scoring placeholder** (future iteration) will sit in a separate micro‑service that can be hooked into the token issuance pipeline; this directly addresses **underserved outcome (a) AI‑driven adaptive verification for RFID/NFC in EV/EVC ecosystems (R9‑a)**.  
  - **ROI analytics dashboard** will expose latency, conversion lift, fraud‑reduction metrics (R9‑b) via a dedicated `/metrics` endpoint, fulfilling the “quantitative ROI benchmarks” request.  
  - **Unified SSO / OAuth2/OIDC support** (addressing community feedback on SAML parser differentials—S35) is built via a pluggable provider interface; the current JWT‑based flow covers the majority of SaaS use‑cases, reducing parser‑dependency.  

### 10.3 Whitespace Opportunities  

- **Whitespace finding (R11)** calls for a modular platform that (i) implements NIST SP 800‑63B‑4, (ii) adds AI‑powered adaptive risk scoring for legacy tokens, and (iii) supplies built‑in ROI analytics.  
- Our architecture **exposes a plug‑in point** (`src/auth/riskScoring.ts`) where an external AI model can be injected; the current baseline implements *static* token‑age logic but the interface is ready for future ML integration.  
- **Latency target (p95 ≤ 200 ms)** directly tackles the conversion‑rate loss described in **R5**, turning a market‑wide performance pain point into a competitive advantage.  

### 10.4 Community Evidence Influence  

- **Hacker News discussion (S35)** highlighted “parser‑differential bypasses of SAML SSO” as a critical bug. Our decision to **avoid SAML parsing** in the core service and instead rely on **JWT‑based OAuth2/OIDC** stems from that concern.  
- **Supabase Auth forum (S36)** requested “unified SSO, mobile‑SDK, and server‑side session handling”. The **single `authService`** module provides both server‑side token validation and a thin client SDK wrapper (will be published as `@myorg/auth-client`), satisfying that community wish.  
- **Authelia/Lldap forum (S37)** and the **Ask HN thread (S38)** stressed the need for **role‑based denial‑by‑default** and **strong password hashing**. Accordingly, `rbacMiddleware.ts` implements a deny‑by‑default policy, and `crypto.ts` enforces bcrypt cost ≥ 12 (aligned with NIST 800‑53).
