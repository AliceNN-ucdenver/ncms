## Problem Statement and Scope  

**Problem:** The identity‑service authentication market is expanding rapidly (**R1, R2, R3**), yet adoption of **phishing‑resistant, NIST‑compliant authentication** remains low (**R5, R3**), creating a security‑gap that leads to frequent identity‑related breaches (**R5, R6**).  

**Scope – In Scope:**  
- Design of **authentication patterns** (passwordless, MFA, federated SSO) for a cloud‑native identity service.  
- Implementation of **NIST SP 800‑63B‑4 AAL3 compliance** (phishing‑resistant methods, password length, non‑exportable authenticator).  
- Integration with **OpenID Connect / SAML** in a multi‑cloud CI/CD pipeline (**R7**).  
- Achievement of **measurable ROI/latency benefits** akin to the **Azure VMware Solution case study** (**R8**).  

**Scope – Out of Scope:**  
- Development of UI components for user‑facing sign‑up flows (excluded because UI is not covered by the retrieved expert advice).  
- Long‑term data‑archival strategies beyond the 2‑year GDPR retention policy (excluded as no explicit research source addresses archival beyond ROI).

## Security Requirements with Threat Tracing  

| Threat | Control | Expert Source | Standard | Implementation |
|--------|---------|---------------|----------|----------------|
| **THR‑001 – Spoofing (JWT validation weakness)** | Token revocation list, short‑lived access tokens, rotating RSA‑2048+ keys | E2 (THR‑001) | NIST 800‑63B (cryptographic authenticator) | JWT middleware validates signature, checks revocation list, enforces ≤ 15 min lifetime; key rotation script every 90 days. |
| **THR‑002 – Tampering (NoSQL injection)** | Strict JSON schema validation, driver‑level parameter binding, least‑privilege MongoDB roles | E2 (THR‑002) | OWASP A03:2021 (Injection) | Validation layer rejects any field not in allow‑list; all DB calls use parameterized queries; DB user has only `readWrite` on intended collection. |
| **THR‑003 – Repudiation** | Signed audit logs with hash‑chain, immutable storage | E2 (logging) | NIST 800‑63B (identity proofing) | Each auth event appended to append‑only log, signed with HMAC‑SHA‑256; log retention 2 years. |
| **THR‑004 – Information Disclosure (PII exposure)** | Mask PII in logs, encrypt data at rest, GDPR‑compliant API | E2 (GDPR alignment) | GDPR Art. 5‑6, NIST 800‑63B (identity proofing) | Log fields `email`, `ip_address` truncated; encryption keys managed via KMS; API returns only required fields. |
| **THR‑005 – Elevation of Privilege** | RBAC/ABAC checks on every write, MFA for admin | E2 (MFA, broken access control) | OWASP A01:2021 (Broken Access Control) | every API write verifies requester owns the resource; admin roles require MFA; access control matrix enforced centrally. |
| **THR‑006 – Denial‑of‑Service** | Rate limiting (X req/min), exponential back‑off, circuit breaker | E2 (rate limiting) | NIST 800‑63B (availability) | API gateway enforces 1000 req/min per client; on consecutive failures, back‑off doubles delay up to 30 s. |
