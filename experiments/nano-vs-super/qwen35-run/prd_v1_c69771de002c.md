# Create a PRD for: Authentication patterns for identity services (research_id: 23784d56a2b2) (project_id: PRJ-963bc59e) — Product Requirements Document

## Input Premises
State what each input source establishes.

### Research Premises
- **R1**: Research establishes: The Identity-as-a-Service market is valued between USD 7.69 billion and USD 16.2 billion in 2025, indicating high commercial demand for scalable identity solutions. — section: Source Document [S1, S2]
- **R2**: Research establishes: NIST SP 800-63-4 is the governing standard effective August 2025, superseding version 3 and defining digital identity assurance levels, excluding location data as primary factors. — section: Source Document [S6, S7, S8, S9]
- **R3**: Research establishes: OIDC is becoming the default protocol for new mobile/API integrations, while SAML serves legacy workforce systems. — section: Source Document [S15]
- **R4**: Research establishes: Identity exposure data grew 22% annually to 53.3 billion records, with AI-driven phishing being a primary vector. — section: Source Document [S12, S14]
- **R5**: Research establishes: Board-level ROI metrics require loss-avoidance math and probability-weighted loss per service rather than simple activity throughput. — section: Source Document [S20]

### Expert Premises
- **E1**: Architect recommends: The system implements JSON Web Tokens (JWT) for stateless authentication with Role-Based Access Control (RBAC) embedded in the token payload. — evidence: ADR-003
- **E2**: Security identifies: THR-001 (Spoofing) risk exists due to potential JWT forgery and weak validation, requiring Token Revocation Lists. — evidence: STRIDE Model
- **E3**: Security identifies: THR-002 (Tampering) risk exists due to insufficient input validation leading to NoSQL injection. — evidence: STRIDE Model
- **E4**: Security identifies: Current controls (bcrypt hashing, session management) are marked `not_started` and must be aligned with NIST SP 800-63B and OWASP A01:2021. — evidence: Compliance & Implementation Requirements

## Problem Statement and Scope
Ground in research premises. Cite R[N] or E[N] for each claim.

**In Scope**: 
- Designing identity service patterns that adhere to NIST SP 800-63-4 guidelines (R2) to ensure compliance.
- Integrating OIDC protocols for new mobile/API channels while supporting SAML for legacy workforce (R3).
- Mitigating Spoofing (E2) and Tampering (E3) threats through short-lived JWTs and input validation.
- Implementing loss-avoidance ROI metrics in the analytics dashboard (R5).
- Supporting JWT-based stateless architecture with RBAC embedded in claims (E1).

**Out of Scope**: 
- Replacing the entire Service-Oriented Architecture (SOA) foundation (ADR-001).
- Implementing hardware root of trust beyond standard software tokens (Evidence Gap).
- Direct hardware-based authentication factors excluded by NIST 63-4 (R2).

## Goals and Non-Goals

### Goals
1.  Achieve NIST SP 800-63-4 compliance by August 2025 — traced to R2, E4 because both mandate adherence to the updated digital identity standard.
2.  Reduce Spoofing risk (THR-001) to residual Medium or lower — traced to E2 because the STRIDE analysis identifies token validation as the critical weakness.
3.  Implement OIDC as the default protocol for new integrations — traced to R3, E1 because market trends favor OIDC and the project uses JWT stateless patterns.
4.  Display loss-avoidance metrics on executive dashboards — traced to R5, E4 because board expectations require probability-weighted loss data.

### Non-Goals
1.  Implementing hardware-based authentication factors — excluded because research indicates location/device identity are not primary authentication factors (R2).
2.  Maintaining SAML as the primary protocol for new mobile clients — excluded because expert architecture utilizes JWT stateless patterns and research favors OIDC (R3, E1).
3.  Achieving real-time identity intelligence for B2B sales conversion — excluded because this PRD focuses on security and architecture, not sales funnel conversion metrics (S22).

## Architecture Alignment
Each decision must trace to an expert premise:
- **Service-Oriented Architecture (SOA)**: The system is built on SOA governed by CALM specifications — recommended by E1 based on ADR-001, ensuring automated governance.
- **Document-Oriented Storage (MongoDB)**: Movie metadata and user reviews stored in MongoDB with Mongoose — recommended by E1 based on ADR-002 for read-heavy scalability.
- **JWT Stateless Pattern**: Authentication utilizes JWT with inline RBAC — recommended by E1 based on ADR-003 to minimize external IdP dependencies.
- **In-Memory Isolation**: Testing uses `mongodb-memory-server` for ephemeral instances — recommended by E1 based on ADR-004 for CI/CD parallelism.
